# 개발 노트

채점 항목인 과제 분석, AI 활용, 가장 어려웠던 기능에 대한 답과, 작업하면서 내린 설계 결정을 정리했습니다.

## 1. 과제를 어떻게 분석했는가

요구사항을 그대로 구현하기 전에 이 과제가 무엇을 보려는지부터 읽었습니다. 명세와 시나리오에서 다음을 확인했습니다.

- 구현의 양은 배점이 낮다고 명시되어 있다.
- 시나리오에서 "지속적으로 확장 개발 가능해야 한다", "시연의 목표는 API로 AI를 활용할 수 있다"를 강조한다.
- "향후 자사의 대외비 문서를 학습시키고 싶어 한다"는 요건이 있다.
- API 키는 따로 주지 않는다.
- 채점 질문 세 개가 모두 문서로 설명하는 항목이다.

그래서 기능을 빠짐없이 채우기보다, 핵심을 확장 가능한 구조로 만들고 그 판단 근거를 설명하는 쪽이 이 과제의 목표라고 봤습니다.

가장 신경 쓴 부분은 LLM 공급자 추상화입니다. "API로 AI 활용", "확장성", "대외비 문서(RAG)", "키 미제공"이 모두 한 곳을 가리켰습니다. `LlmProvider` 인터페이스를 두고 그 아래에 실제 연동용 `OpenAiProvider`와 키가 없을 때 쓰는 `MockProvider`를 두었습니다. 어떤 구현을 쓸지는 `LlmConfig`가 키 유무로 정하므로, 평가자가 키 없이 `./gradlew bootRun`만으로 전체 기능을 시연할 수 있습니다.

출제사가 공개한 [`sionic-ai/spring-ai-tutorial`](https://github.com/sionic-ai/spring-ai-tutorial)이 Kotlin과 Spring Boot, Spring AI로 LLM 호출과 RAG 챗봇을 다루고, 회사 본업이 기업용 RAG 플랫폼이라는 점도 확인했습니다. 과제의 "대외비 문서 학습" 요건이 이 회사 제품과 맞닿아 있다는 뜻이라, 추상화에 집중한 판단에 더 확신을 가졌습니다. `LlmProvider`는 나중에 Spring AI의 `ChatClient`나 RAG 구현이 들어갈 수 있는 자리이기도 합니다.

시간은 스캐폴딩, 인증/JWT, 대화(스레드 세션과 스트리밍), 피드백, 분석/CSV, 문서 순으로 배분했습니다. 인증과 대화, 스트리밍에 깊이를 두고 피드백과 분석은 요구사항을 충족하는 수준으로 처리했습니다.

명세에 없지만 두 가지를 추가했습니다.

- `ActivityLog`: 로그인 기록은 다른 테이블에 남지 않습니다. 24시간 활동 집계를 한 쿼리로 처리하려고 이벤트 로그를 두었습니다.
- `ChatThread.last_chat_at`: 30분 세션 판정을 매번 `max(chat.created_at)`로 조회하지 않고 컬럼 하나로 처리하려고 두었습니다.

## 2. AI를 어떻게 활용했고, 어떤 어려움이 있었는가

Claude Code를 사용했고, 방향은 직접 정하고 반복적인 코드 작성을 맡기는 식으로 썼습니다.

먼저 코드를 쓰기 전에 DB(H2), 키가 없을 때의 대응(Mock 폴백), 패키지 구조, ERD, 시간 배분을 먼저 정했습니다. 방향을 정해두니 중간에 되돌리는 일이 거의 없었습니다. 엔티티나 DTO 같은 반복 코드는 빠르게 만들고, 아낀 시간을 세션 로직, 스트리밍, 트랜잭션 경계처럼 판단이 필요한 부분에 썼습니다. 작성한 다음에는 컴파일, 서버 기동, curl 테스트를 짧게 반복했습니다. 회원가입부터 스트리밍, CSV, 권한 거부(401/403)까지, 그리고 실제 OpenAI로 멀티턴 대화("방금 말한 그 도시는?"에 직전 답을 기억하는지)까지 응답을 직접 확인했습니다.

가장 어려웠던 점은, AI가 만든 코드가 컴파일도 되고 얼핏 맞아 보이는데 런타임 동작이 틀린 경우였습니다. 3번에서 설명하는 트랜잭션 self-invocation이 그런 사례입니다. 정적으로는 문제가 없어서 동작 원리를 알고 의심해야 발견됩니다. 그래서 두 가지를 지켰습니다. 하나는 경계를 직접 정하는 것입니다. 예를 들어 SSE 처리에서 자칫 전체를 WebFlux로 바꾸는 방향으로 갈 수 있었는데, MVC를 유지하고 외부 호출만 WebClient로 한다는 선을 정해두니 그 안에서 작업이 맞아 들어갔습니다. 다른 하나는 동작을 직접 확인하는 것입니다. 될 것 같다는 판단 대신, 실제로 기동해서 토큰이 흐르고 done 이벤트에 저장된 대화가 담기는지 확인했습니다.

## 3. 가장 어려웠던 기능: 스트리밍 응답과 저장, 트랜잭션 경계

대화 생성의 스트리밍 모드가 가장 까다로웠습니다. 세 가지가 얽혀 있습니다.

먼저 한 엔드포인트가 JSON과 SSE를 모두 반환해야 합니다. `isStreaming` 값에 따라 같은 `POST /api/chats`가 `application/json` 또는 `text/event-stream`으로 응답해야 합니다. 컨트롤러 반환 타입을 `Any`로 두고 실제 반환값이 `SseEmitter`인지 DTO인지에 따라 Spring이 처리하게 했는데, 이론상 모호한 부분이라 양쪽을 직접 호출해서 확인한 뒤에 안전하다고 판단했습니다.

다음으로, 토큰은 흘려보내되 완성된 답변만 저장해야 합니다. `streamChat()`이 반환하는 `Flux<String>`을 구독해서 토큰을 즉시 SSE로 내보내면서 `StringBuilder`에 모으고, `onComplete` 시점에 전체 답변을 한 번만 저장한 뒤 done 이벤트로 최종 대화를 전달합니다.

마지막이 실제로 가장 까다로웠던 부분입니다. `Flux` 구독 콜백은 요청 스레드가 아니라 다른 스레드에서 실행됩니다. 처음에는 저장 로직을 같은 서비스의 `@Transactional` 메서드로 두었는데, 이건 self-invocation이라 Spring AOP 프록시를 거치지 않아 트랜잭션이 적용되지 않습니다. 컴파일도 되고 얼핏 동작해 보여서 놓치기 쉬운 종류입니다. 트랜잭션이 필요한 `prepareThread`(스레드 세션 해석)와 `persistAnswer`(저장)를 별도 빈 `ChatStore`로 분리해서, 외부 호출을 통해 프록시가 정상 적용되도록 했습니다. 동기와 스트리밍 두 경로가 같은 메서드를 공유합니다.

```
ChatService.createChatStream()
  └─ chatStore.prepareThread()      // 트랜잭션 (스레드 세션, 이전 대화 로드)
  └─ llmProvider.streamChat(...)    // 비트랜잭션, 외부 LLM 호출 (다른 스레드)
        ├─ onNext  : 토큰 누적, SSE 전송
        └─ onComplete: chatStore.persistAnswer()  // 트랜잭션 (별도 빈, 프록시 정상)
```

이렇게 나누면 외부 LLM 호출이 도는 동안 DB 트랜잭션을 잡고 있지 않아도 된다는 이점도 있습니다.

그다음으로 어려웠던 건 30분 세션 경계였습니다. `prepareThread(userId, now)`가 시각을 파라미터로 받게 해서 `ChatSessionTest`에서 이내, 정확히 30분, 초과 케이스를 시간에 의존하지 않고 검증했습니다.

## 4. 설계 결정과 한계

코드 양을 늘리는 대신 무엇을 하고 무엇을 하지 않을지 정했습니다.

적용한 것은 다음과 같습니다.

- 이벤트 기반 모듈 분리: 스레드를 삭제하면 그 대화를 참조하던 피드백도 정리해야 하는데, `ChatService`가 `FeedbackRepository`를 직접 의존하면 chat과 feedback 패키지가 서로 참조하게 됩니다. 대신 `ThreadDeletedEvent`를 발행하고 피드백 쪽(`FeedbackCleanupListener`)이 구독해서 정리합니다.
- 방어 처리: 내부 예외 메시지 비노출, DB 유니크 위반 시 409 반환, CSV 수식 인젝션 방어, 페이지 size 상한, 스트리밍 구독 정리, OpenAI 호출 타임아웃과 에러 매핑.
- Swagger(OpenAPI): 스펙을 모르는 고객사에 문서 형태로 전달하라는 시나리오에 맞춰 추가했습니다. `/swagger-ui.html`에서 JWT 인증 후 전체 API를 바로 시연할 수 있습니다.

의도적으로 하지 않은 것(한계)은 다음과 같습니다.

- LLM 호출 실패 시 빈 스레드가 남을 수 있습니다. 외부 호출 동안 트랜잭션을 잡지 않으려고 스레드 해석과 저장을 분리한 결과, 첫 질문에서 LLM이 실패하면 대화 없는 스레드가 남습니다. 인지하고 있으며 보상 로직(정리 배치나 실패 시 롤백)은 범위상 넣지 않았습니다.
- JWT 만료와 위조를 구분하지 않습니다. 파싱 실패를 모두 401로 처리합니다. 리프레시 토큰을 도입한다면 만료만 구분하겠지만, 지금은 공격자에게 단서를 주지 않는 면에서도 단순하게 두는 편이 낫다고 봤습니다.
- 테스트는 100% 커버리지 대신 가장 깨지기 쉬운 30분 세션 경계에 집중했습니다.

Spring AI를 쓰지 않고 `LlmProvider`를 직접 만든 이유는, 추상화를 직접 보여주고 싶었고 무거운 의존성 없이 스트리밍과 에러, 모델 선택을 직접 다루고 싶었기 때문입니다. 다만 이 인터페이스는 내부 구현을 Spring AI `ChatClient`로 바꿀 수 있는 자리이기도 합니다.

RAG로의 확장은 새 `LlmProvider` 구현 하나로 붙습니다. 질문을 임베딩하고 벡터 스토어에서 사내 문서를 검색해 그 컨텍스트를 messages에 넣은 뒤, 기존과 같은 `chat()` / `streamChat()`을 호출하면 됩니다. `ChatService`와 외부 API, 인증, 기록, 피드백, 분석은 바뀌지 않고, 교체 지점은 `LlmConfig` 한 곳입니다.
