# 개발 노트

채점 기준 3개 질문(과제 분석 / AI 활용 / 가장 어려웠던 기능)에 대한 답과, 의식적으로 내린 설계 결정을 정리했습니다.

---

## 1. 과제를 어떻게 분석했는가

요구사항을 구현하기 전에 **이 과제가 무엇을 평가하려는지**부터 읽었습니다. 명세와 시나리오에서 다음 신호를 뽑았습니다.

- "구현의 **양은 배점이 낮다**"
- 시나리오: "**지속적으로 확장 개발 가능**해야 한다", "시연 목표 = API 로 AI 를 활용할 수 있다"
- "향후 자사의 **대외비 문서를 학습**시키고 싶어 한다"
- "API KEY 는 별도로 전달되지 않는다"
- 채점 질문 3개 중 3개가 **문서/설명**

이 신호들이 한 점으로 수렴했습니다. **기능을 빠짐없이 채우는 게 아니라, 핵심을 확장 가능한 구조로 만들고 판단 근거를 설명하는 것**이 이 과제의 목표라고 결론지었습니다.

### 핵심 베팅 — LLM 공급자 추상화
"API 로 AI 활용 / 확장성 / 대외비 문서(RAG) / 키 미제공"이 모두 한 지점을 가리켰습니다. 그래서 `LlmProvider` 인터페이스 하나에 집중했습니다.

- `OpenAiProvider` — 실제 연동
- `MockProvider` — 키가 없을 때 자동 폴백 → **평가자가 키 없이 `./gradlew bootRun` 만으로 전 기능 시연 가능**
- 활성 구현은 `LlmConfig` 가 키 유무로 자동 선택 (코드 변경 0)

### 출제사 리서치로 의도를 재확인
출제사가 공개한 [`sionic-ai/spring-ai-tutorial`](https://github.com/sionic-ai/spring-ai-tutorial) 이 **Kotlin + Spring Boot + Spring AI 로 LLM 호출 → RAG 챗봇**을 다루고, 회사 본업이 **기업용 RAG 플랫폼**임을 확인했습니다. 즉 과제의 *"대외비 문서 학습"* 한 줄은 이 회사의 핵심 제품(RAG)을 가리킨 것이었습니다.

→ 이 발견이 추상화 베팅의 확신을 키웠습니다. `LlmProvider` 는 **Spring AI 의 `ChatClient` 나 RAG Advisor 가 그대로 들어갈 수 있는 동일한 seam** 입니다. (확장 경로는 4번 참고)

### 시간 배분
스캐폴딩 → 인증/JWT → **대화(스레드 세션 + LLM 추상화 + 스트리밍)** → 피드백 → 분석/CSV → 문서. 핵심 3개(인증·대화·스트리밍)에 깊이를 두고, 피드백/분석은 요구사항을 정확히 충족하는 수준으로 빠르게 처리했습니다.

### 명세에 없지만 추가한 것
- `ActivityLog`: **로그인 횟수는 어느 테이블에도 남지 않습니다.** 24h 활동 집계를 단일 쿼리로 처리하려고 이벤트 로그를 도입했습니다.
- `ChatThread.last_chat_at`: 30분 세션 판정을 매번 `max(chat.created_at)` 조회 대신 컬럼 하나로.

## 2. AI(Claude Code)를 어떻게 활용했고, 어떤 어려움이 있었는가

### 활용 방식 — "설계는 내가, 타이핑은 AI"
- **전략을 먼저 합의했습니다.** 코드 한 줄 쓰기 전에 DB 선택(H2), 키 부재 대응(Mock 폴백), 패키지 구조, ERD, 시간 배분을 확정했습니다. 방향을 고정한 뒤 구현에 들어가니 되돌아가는 비용이 거의 없었습니다.
- **보일러플레이트는 AI 로 가속**하고, 아낀 시간을 세션 로직·스트리밍·트랜잭션 경계 같은 **판단이 필요한 곳**에 썼습니다.
- **짧은 검증 루프**: 작성 → 컴파일 → 실제 서버 기동 → curl 스모크 테스트를 반복했습니다. 회원가입부터 스트리밍·CSV·권한 거부(401/403)까지, 그리고 실제 OpenAI 멀티턴("방금 말한 그 도시는?" → 직전 답변을 기억)까지 실제 응답으로 확인했습니다.

### 어려움 — AI 는 "컴파일되지만 틀린 코드"를 만든다
가장 큰 어려움은, AI 가 만든 코드가 **컴파일도 되고 얼핏 맞아 보이는데 런타임 의미가 틀린** 경우였습니다. 3번에서 설명하는 **트랜잭션 self-invocation** 이 정확히 그 사례입니다. 정적으로는 아무 문제가 없어서, **동작 원리를 알고 의심해야만** 발견됩니다.

그래서 두 가지 원칙을 지켰습니다.
1. **경계는 사람이 정한다.** 예컨대 SSE 처리에서 자칫 전면 WebFlux 전환으로 흐를 수 있었는데, "MVC 유지 + SSE + 외부 호출만 WebClient" 라는 경계를 제가 못박아야 AI 가 그 안에서 정확히 일했습니다.
2. **신뢰하되 검증한다.** "돌 것 같다"가 아니라 실제 기동해 SSE 토큰이 흐르고 `done` 이벤트에 영속화된 대화가 담기는지 눈으로 확인했습니다.

## 3. 가장 어려웠던 기능 — 스트리밍 응답 + 영속화 + 트랜잭션 경계

**대화 생성의 스트리밍 모드**가 가장 까다로웠습니다. 세 가지가 한꺼번에 얽힙니다.

### (1) 한 엔드포인트가 JSON 과 SSE 를 모두 반환
`isStreaming` 에 따라 같은 `POST /api/chats` 가 `application/json`(동기) 또는 `text/event-stream`(스트리밍)을 반환해야 합니다. 컨트롤러 반환 타입을 `Any` 로 두고 실제 반환값(`SseEmitter` vs DTO)에 따라 Spring 이 처리하도록 했는데, 이론상 모호한 부분이라 **실제 기동해 양쪽을 curl 로 확인**하고서야 안전하다고 판단했습니다.

### (2) "토큰은 흘려보내되, 완성된 답변만 저장"
`streamChat()` 이 돌려주는 `Flux<String>` 을 구독해 delta 를 즉시 SSE 로 내보내면서 `StringBuilder` 에 모으고, `onComplete` 시점에 전체 답변을 **1번만** 영속화한 뒤 `done` 이벤트로 최종 대화를 전달합니다.

### (3) 진짜 함정 — 트랜잭션 self-invocation
`Flux` 구독 콜백은 **요청 스레드가 아닌 다른 스레드**에서 실행됩니다. 처음엔 영속화 로직을 같은 서비스의 `@Transactional` 메서드로 두었는데, 이는 **self-invocation** 이라 Spring AOP 프록시를 우회해 트랜잭션이 걸리지 않습니다. **컴파일도 되고 얼핏 동작해 보여서 가장 위험한 종류의 버그**입니다.

해결: 트랜잭션 경계가 필요한 `prepareThread`(스레드 세션 해석)와 `persistAnswer`(영속화)를 **별도 빈 `ChatStore`** 로 분리해, 외부 호출을 통해 프록시가 정상 적용되도록 했습니다. 동기/스트리밍 두 경로가 같은 트랜잭션 안전 메서드를 공유합니다.

```
ChatService.createChatStream()
  └─ chatStore.prepareThread()      // 트랜잭션 (스레드 세션 + 이전 대화 로드)
  └─ llmProvider.streamChat(...)    // 비트랜잭션, 외부 LLM 호출 (다른 스레드)
        ├─ onNext  : delta 누적 + SSE 송신
        └─ onComplete: chatStore.persistAnswer()  // 트랜잭션 (별도 빈 → 프록시 정상)
```

이 분리는 부수적으로 **외부 LLM 호출(수 초) 동안 DB 트랜잭션을 잡지 않게** 해주는 이점도 있습니다.

> 다음으로 어려웠던 것은 30분 스레드 세션 경계였고, `prepareThread(userId, now)` 가 시각을 주입받게 설계해 `ChatSessionTest` 에서 이내/정확히 30분/초과를 결정적으로 검증했습니다.

## 4. 의식적으로 내린 설계 결정과 트레이드오프

코드 양을 늘리는 대신, **무엇을 하고 무엇을 하지 않을지 명시적으로 선택**했습니다.

### 채택한 것
- **이벤트 기반 모듈 분리**: 스레드 삭제 시 연관 피드백을 정리해야 하는데(고아 방지), `ChatService` 가 `FeedbackRepository` 를 직접 의존하면 chat↔feedback 패키지 순환이 생깁니다. 대신 `ThreadDeletedEvent` 를 발행하고 피드백 모듈이 구독하게 해 순환 없이 정리합니다. (`FeedbackCleanupListener`)
- **방어적 처리**: 내부 예외 메시지 비노출(정보 누출 방지), DB 유니크 위반 race → 409, CSV 수식 인젝션 방어, 페이지 size 상한, 스트리밍 구독 누수 방지, OpenAI 호출 타임아웃/에러 매핑.
- **Swagger(OpenAPI)**: 시나리오의 "API spec 이해가 없는 고객사에 메뉴얼과 함께 전달"에 직접 대응. `/swagger-ui.html` 에서 JWT 인증 후 전 API 를 바로 시연할 수 있습니다.

### 의도적으로 하지 않은 것 (한계점)
- **LLM 호출 실패 시 빈 스레드 잔존**: 외부 호출 동안 트랜잭션을 잡지 않으려고 스레드 해석/영속화를 분리한 결과, 첫 질문에서 LLM 이 실패하면 대화 없는 스레드가 남을 수 있습니다. 인지하고 있으며, 보상 로직(정리 배치/실패 시 롤백)은 데모 범위상 제외했습니다.
- **JWT 만료 vs 위조 미구분**: 파싱 실패를 모두 동일하게 401 처리합니다. 리프레시 토큰을 도입한다면 만료만 구분하겠지만, 현재는 공격자에게 단서를 주지 않는 정보 최소화 측면에서도 합리적이라 단순하게 두었습니다.
- **테스트 범위**: "구현 양 배점이 낮다"는 신호에 따라 100% 커버리지 대신 **가장 깨지기 쉬운 30분 세션 경계**에 집중했습니다.

### Spring AI 를 쓰지 않은 이유
출제사가 Spring AI 튜토리얼을 운영함을 알고도 직접 `LlmProvider` 를 만들었습니다. (a) 추상화를 1차 원리로 보여주고 싶었고, (b) 무거운 의존성 없이 스트리밍·에러·모델 선택을 완전히 제어하고 싶었기 때문입니다. 다만 이 인터페이스는 **언제든 내부 구현을 Spring AI `ChatClient` 로 갈아끼울 수 있는 동일한 자리**입니다.

### RAG 로의 확장 경로 (대외비 문서 학습)
요구된 "향후 대외비 문서 학습"은 이 구조에서 새 `LlmProvider` 구현 하나로 자연스럽게 붙습니다.

```
RagProvider : LlmProvider
  1) 질문 임베딩 → 2) 벡터 스토어에서 사내 문서 검색
  3) 검색된 컨텍스트를 messages 에 주입 → 4) 기존과 동일하게 chat()/streamChat()
```

`ChatService` 와 외부 REST 계약, 인증·기록·피드백·분석은 **전혀 바뀌지 않습니다.** 교체 지점은 `LlmConfig` 한 곳입니다.
